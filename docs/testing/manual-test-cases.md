# 手工测试用例

## 失败诊断导出

### 前置条件

- 已加载开发版扩展
- 打开任一支持的平台页面
- 选择至少 1 个文件，确保可以进入批量重命名面板

### 用例 1：失败后出现导出入口

1. 构造一个必然失败的重命名场景。
2. 执行批量重命名，等待结果页出现。
3. 确认结果统计里 `失败` 大于 `0`。
4. 确认左侧结果区出现诊断提示条，包含 `导出诊断` 和 `稍后处理`。

期望结果：

- 提示条只在存在失败项时出现
- 提示条展示失败后反馈引导文案

### 用例 2：导出诊断文件

1. 在失败结果页点击 `导出诊断`。
2. 等待浏览器下载 JSON 文件。
3. 打开下载的文件，确认文件名符合 `cloud-drive-renamer-diagnostic-YYYY-MM-DD-HHmm.json`。
4. 检查 JSON 至少包含以下字段：
   - `exportedAt`
   - `lastFailure.summary`
   - `lastFailure.failures`
   - `logs`

期望结果：

- 导出成功后，提示条切换为反馈动作态
- 提示条展示导出的诊断文件名

### 用例 3：导出后的反馈动作

1. 在导出成功态点击 `GitHub`。
2. 确认打开 GitHub issue 新建页，正文包含平台、扩展版本、失败数量、导出时间、诊断文件名。
3. 返回扩展后点击 `邮件`。
4. 确认打开默认邮件客户端，主题和正文已带上诊断说明。
5. 返回扩展后点击 `复制说明`，将内容粘贴到任意文本框中确认文案完整。

期望结果：

- 三个动作都不会让提示条消失
- 复制说明至少包含平台、扩展版本、失败数量、导出时间、诊断文件名

### 用例 4：稍后处理

1. 在 `ready` 或 `exported` 状态点击 `稍后处理`。
2. 确认当前结果页中的诊断提示条消失。
3. 点击 `重试` 或 `返回`，确认原有执行流程不受影响。

期望结果：

- 诊断提示条隐藏
- `重试`、`返回`、`撤销` 等既有动作可继续正常使用

### 用例 5：重试后恢复

1. 在存在失败项的结果页点击 `重试`。
2. 若重试后全部成功，确认诊断提示条不再出现。
3. 若重试后仍有失败，确认提示条重新回到 `导出诊断` 状态。

期望结果：

- 全部成功时隐藏提示条
- 仍有失败时重新显示导出入口
